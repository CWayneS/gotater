How to use Git Playground

A visual simulation. Type commands and watch files move between zones.

Create & Edit Files

  • touch filename — create a new file in working directory
  • edit filename — modify an existing file (marks it as changed)
  • rm filename — delete a file from working directory

Core Git Flow

  • git add filename or git add . — stage files (Working → Staging)
  • git commit -m "msg" — commit staged files (Staging → Local Repo)
  • git push — push commits to remote (Local → Remote)
  • git pull — pull from remote (Remote → Working + Local)

Inspect & Undo

  • git status — see what's changed and staged
  • git log — view commit history
  • git diff — show unstaged changes
  • git stash — temporarily shelve changes
  • git stash pop — restore shelved changes
  • git reset filename — unstage a file (Staging → Working)
  • git checkout -- filename — discard working changes

Branching

  • git branch branchname — create a new branch
  • git checkout branchname — switch to a branch
  • git merge branchname — merge another branch into current

🚀 First Push — New Project Setup beginner

You just created a project and want to get it on GitHub for the first time.
1touch index.html — create your files
2touch styles.css
3touch app.js
4git add . — stage everything
5git commit -m "initial commit" — save snapshot
6git push — upload to GitHub

🔄 Daily Driver — Edit, Commit, Push beginner

Your everyday loop: make changes, save them, push them up. This is probably what you do most.
1edit app.js — make changes to an existing file
2git status — see what changed
3git add . — stage changes
4git commit -m "update app logic"
5git push

🎯 Selective Staging — Commit Only Some Files intermediate

You changed several files but only want to commit a few. This keeps your history clean and logical.
1edit index.html — bug fix
2edit styles.css — bug fix
3edit app.js — unrelated experiment
4git add index.html — stage only the fix
5git add styles.css
6git commit -m "fix layout bug" — clean commit!
7git status — app.js still in working dir

📦 Stash & Switch — Pause Work Temporarily intermediate

You're mid-task but need to quickly do something else. Stash shelves your changes so you can come back later.
1edit app.js — you're mid-feature
2git stash — shelve it, working dir is clean
3git status — clean! safe to switch tasks
4git stash pop — bring your changes back
5git status — changes are back, keep working

↩️ Oops, Unstage — Remove from Staging intermediate

You accidentally staged a file you didn't mean to. Move it back to working without losing changes.
1edit index.html
2edit app.js
3git add . — oops, staged everything
4git reset app.js — unstage just app.js
5git status — app.js back in working, index.html still staged
6git commit -m "update index"

🌿 Feature Branch — Build, Then Merge advanced

Work on a feature in isolation, then merge it back to main when it's ready.
1git branch new-feature — create branch
2git checkout new-feature — switch to it
3touch feature.js — build the feature
4git add .
5git commit -m "add feature"
6git checkout main — switch back
7git merge new-feature — bring feature into main
8git push

⬇️ Pull First — Start Your Day beginner

Good habit: always pull before starting work to make sure you have the latest code (especially with collaborators).
1git pull — grab latest from remote
2git log — see what's new
3touch newfile.js — now start your work
4git add .
5git commit -m "add newfile"
6git push

The Conveyor Belt

Think of it as a conveyor belt flowing left to right: you create files in your working directory, stage the ones you want to snapshot, commit that snapshot to your local history, then push it to the shared cloud. pull goes the other way.

Why Staging Exists

Staging is like a loading dock. You pick which boxes (files) go on the truck (commit) before it leaves. Without it, every commit would be "everything I changed" — staging lets you say "just these specific changes go together."

Local vs Remote

Your local repo is your personal history. You can commit all day without anyone seeing it. push is when you share. This means you can experiment, make messy commits, even reset — all before anyone else knows.

Branches Are Parallel Timelines

A branch is a copy of your project's timeline where you can make changes without affecting the original. When you're happy with the result, you merge it back. Think of it like a "save slot" — you branch off, try something, and if it works, bring it home.

Stash Is a Clipboard

Stash temporarily copies your uncommitted changes to a shelf. Your working directory becomes clean (as if you hadn't changed anything). stash pop pastes them back. Useful when you need to quickly switch tasks without committing half-done work.
